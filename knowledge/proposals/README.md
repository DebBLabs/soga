# PROPOSALS

Purpose

This directory contains proposed changes that have not yet become part of the project.

AI collaborators should never silently change architecture.

Instead they should document proposals here for review.

---

# Proposal Template

Title

Date

Author

---

## Problem

Describe the current limitation.

---

## Current Behavior

Describe how the repository currently behaves.

---

## Proposed Change

Describe the recommended modification.

Be explicit.

---

## Benefits

Why should this change be considered?

---

## Risks

What architectural or implementation risks exist?

---

## Affected Components

List the files, engines, or workbenches affected.

---

## Repository Impact

Would this change affect:

- architecture

- implementation

- demonstrations

- documentation

- external integrations

---

## Review Required

Before implementation the proposal should receive:

- Architectural review

- Human approval

- Repository acceptance

Only after acceptance should implementation begin.

Proposed

↓

↓

↓

---

# Guiding Principle

Conversation produces ideas.

Repository proposals preserve ideas.

Accepted proposals become project knowledge.

Rejected proposals remain historical record.

Nothing important should disappear simply because a conversation ended.
